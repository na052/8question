# 8番出口風クイズゲーム 仕様書 v1.0

## 1. 概要

本システムは、Web上でプレイする単発型のクイズゲームである。  
ゲームのテーマは「8番出口のクイズ版」とし、プレイヤーは各問題で以下を同時に回答する。

1. 問題画像に関するクイズの答え
2. 問題画像内に存在する違和感の内容（違和感がある場合のみ）

各問題では、クイズ回答と違和感回答の両方が正しい場合のみ次の問題へ進める。  
どちらか一方でも不正解であれば、その時点で進捗は完全にリセットされる。  
8問連続で正解した場合、ゲームクリアとする。

---

## 2. ゲームルール

### 2.1 基本ルール
- 各問題には以下が存在する
  - 1枚の問題画像
  - 1つのクイズ
  - 違和感が0個または1個
- プレイヤーは以下の2つを同時に入力する
  - クイズ回答
  - 違和感回答
- 回答は送信ボタン押下時にまとめて判定する
- 判定結果が以下を満たした場合のみ正解とする
  - クイズ回答が正しい
  - 違和感の有無に応じて、違和感回答が正しい

### 2.2 不正解条件
以下のいずれかに該当した場合、不正解とする。

1. クイズの答えが誤っている
2. 違和感が存在するのに、違和感回答が空である
3. 違和感が存在するのに、違和感回答が誤っている
4. 違和感が存在しないのに、違和感回答に何か記入されている
5. LLM判定に失敗した場合（APIエラー、タイムアウト、異常レスポンスを含む）

### 2.3 正解時の挙動
- 正解した場合、画面に大きく「x問」と表示する
- x は現在の連続正解数とする
- 3秒後に自動的に次の問題へ遷移する
- 8問目に正解した場合はクリアとし、「クリア!!」画面へ遷移する

### 2.4 不正解時の挙動
- 不正解時は進捗を完全にリセットする
- 画面に大きく「0問」と表示する
- 3秒後に自動的に1問目へ遷移する
- リセット後の出題順は再度ランダムとする

### 2.5 クリア条件
- 8問連続で正解した場合、ゲームクリアとする
- クリア画面には「クリア!!」と「ホームに戻る」ボタンを表示する

---

## 3. 出題仕様

### 3.1 出題方式
- 1プレイで出題する問題数は8問
- 問題は全体問題プールからランダムで選出する
- 1プレイ中に一度正解した問題は、リセットされるまで再出題しない
- 問題総数は8問以上を前提とする
- リセット後は、過去の出題履歴を破棄し、再度全問題からランダム出題を行う

### 3.2 問題の前提
- 各問題は1枚の画像で構成する
- 1問題につき違和感は最大1つまで
- 違和感がない問題も存在する

## 4. 判定仕様

### 4.1 クイズ回答判定
- クイズ回答は完全一致で判定する
- 前後の空白はトリムしてから比較する
- それ以外の表記ゆれは許容しない
  - 大文字小文字の違いは不一致
  - 全角半角の違いは不一致
  - 同義語は不一致

### 4.2 違和感回答判定
- 違和感回答は完全一致ではなく、意味ベースで判定する
- 問題ごとに `anomalyId` を1つ持つ
- LLMには以下の目的だけを担わせる
  - 「プレイヤーの違和感回答が、その問題の anomalyId を指しているかどうか」の判定
- 判定条件
  - 正解キーワードを含んでいる
  - 近いニュアンスを持つ
  - 正解として許容される別表現に該当する
- 判定はLLMに1回だけ問い合わせる
- リトライ、再判定、ブレ対策は行わない
- 判定失敗時は不正解とする

### 4.3 違和感なし問題の扱い
- `hasAnomaly = false` の問題では、違和感回答欄は空でなければならない
- 1文字でも入力されていた場合は不正解とする
- この場合、LLM判定は呼び出さない

## 6. 問題データ仕様

### 6.1 問題データ保存方針
- 問題は後から追加できるようにする
- 問題データはコード直書きではなく、永続データとして保持する
- ユーザのプレイ履歴は保存しない
- 保存対象は問題マスタのみとする

### 6.2 問題1件の必須データ項目
各問題は以下の項目を持つこと。

- problemId
- imageFilename
- quizText
- quizAnswer
- hasAnomaly
- anomalyId
- anomalyAnswerCanonical
- anomalyAnswerAliases
- tags

### 6.3 問題データ項目定義
- `problemId`: 問題ID。自動採番または自動生成する
- `imageFilename`: 表示画像ファイル名
- `quizText`: クイズ文
- `quizAnswer`: クイズ正答
- `hasAnomaly`: 違和感の有無（boolean）
- `anomalyId`: 違和感識別子。違和感がある場合のみ必須
- `anomalyAnswerCanonical`: 違和感の代表正答表現
- `anomalyAnswerAliases`: 違和感の別表現候補の配列
- `tags`: 問題分類タグの配列

### 6.4 問題データの補足ルール
- `hasAnomaly = false` の場合:
  - `anomalyId` は null 可
  - `anomalyAnswerCanonical` は空文字または null 可
  - `anomalyAnswerAliases` は空配列
- `hasAnomaly = true` の場合:
  - `anomalyId` は必須
  - `anomalyAnswerCanonical` は必須
  - `anomalyAnswerAliases` は1件以上推奨

### 6.5 タグ自動割り当てルール
管理画面ではタグを手入力させない。  
タグはサーバ側で自動付与する。

初期実装では最低限以下のタグを自動付与する。
- `has_anomaly:true` または `has_anomaly:false`
- `created_from:admin`
- `created_date:YYYY-MM-DD`

補足:
- 将来的に意味的タグの自動生成へ拡張可能
- 初期実装では出題ロジックにタグは使用しない

### 6.6 画像仕様
- 問題画像は1枚画像
- Googleスライド風の横長比率とする
- 初期実装では 16:9 を標準とする
- 推奨形式は PNG
- 画像はローカル保存またはサーバ上のアップロード保存とする

### 6.7 問題データ例
{
  "problemId": "q_0001",
  "imageFilename": "q_0001.png",
  "quizText": "このスライドで主張されている内容を答えよ",
  "quizAnswer": "クラウド移行",
  "hasAnomaly": true,
  "anomalyId": "title_typo",
  "anomalyAnswerCanonical": "タイトルの誤字",
  "anomalyAnswerAliases": [
    "タイトルが誤字",
    "見出しの誤字",
    "タイトルのスペルミス"
  ],
  "tags": [
    "has_anomaly:true",
    "created_from:admin",
    "created_date:2026-04-17"
  ]
}

## 7. 問題追加機能仕様

### 7.1 概要
- 問題追加用の専用ページを作成する
- 管理者が画像と必要項目を入力し、問題を追加できるようにする
- 問題IDとタグは自動割り当てとする

### 7.2 問題追加画面の入力項目
入力項目:
- 問題画像アップロード
- クイズ文
- クイズ正答
- 違和感の有無（true / false）
- 違和感の正解
- 違和感の別表現候補（複数入力可能）

自動生成項目:
- 問題ID
- タグ

### 7.3 問題ID生成ルール
- 問題IDはサーバ側で自動生成する
- 初期実装では連番形式 `q_0001`, `q_0002`, ... を採用する
- 重複してはならない

### 7.4 バリデーション
- 画像が未選択の場合は登録不可
- クイズ文が空の場合は登録不可
- クイズ正答が空の場合は登録不可
- `hasAnomaly = true` の場合、違和感の正解は必須
- `hasAnomaly = false` の場合、違和感正解および別表現候補は空にする
- 別表現候補は重複を排除する

### 7.5 管理ページの公開方針
- ゲーム本体はログイン不要で公開する
- 問題追加ページは一般公開しない
- 初期実装では簡易な管理者保護を設ける
  - 例: ベーシック認証または環境変数パスワード
- 認証方式の詳細は実装時に簡易構成でよい

## 8. API仕様

### 8.1 基本方針
- Web公開する
- LLMはAPIサーバ化する
- 同時接続は10人程度を想定する
- ユーザログインは不要
- セッション保存やスコア保存は不要

### 8.2 プレイ状態管理
- プレイ履歴はサーバに保存しない
- 現在のプレイ状態はフロントエンド側で保持する
- 必要な状態は API リクエスト時に送る
- 状態例:
  - currentStreak
  - usedProblemIds
  - currentProblemId

### 8.3 エンドポイント一覧

#### GET /api/problems/next
次に出題する問題を返す。

リクエストクエリ例:
- usedProblemIds= q_0001,q_0003

レスポンス例:
{
  "problemId": "q_0005",
  "imageUrl": "/uploads/q_0005.png",
  "quizText": "このスライドの内容を答えよ",
  "questionNumber": 3
}

選定ルール:
- 全問題から `usedProblemIds` を除外する
- 残りからランダムに1件選ぶ

#### POST /api/answer
回答判定を行う。

リクエスト例:
{
  "problemId": "q_0005",
  "quizAnswer": "クラウド移行",
  "anomalyAnswer": "タイトルの誤字",
  "currentStreak": 2,
  "usedProblemIds": ["q_0001", "q_0003"]
}

レスポンス例（正解）:
{
  "result": "correct",
  "quizCorrect": true,
  "anomalyCorrect": true,
  "nextStreak": 3,
  "transition": "success"
}

レスポンス例（不正解）:
{
  "result": "incorrect",
  "quizCorrect": true,
  "anomalyCorrect": false,
  "nextStreak": 0,
  "transition": "reset"
}

レスポンス例（クリア）:
{
  "result": "clear",
  "quizCorrect": true,
  "anomalyCorrect": true,
  "nextStreak": 8,
  "transition": "clear"
}

#### POST /api/admin/problems
新規問題を登録する。

送信内容:
- multipart/form-data
- image file
- quizText
- quizAnswer
- hasAnomaly
- anomalyAnswerCanonical
- anomalyAnswerAliases

レスポンス:
- 生成された problemId
- 保存結果

#### GET /api/admin/problems
登録済み問題一覧を返す。

#### DELETE /api/admin/problems/:problemId
問題削除用。初期実装では任意。未実装でも可。

## 9. LLM判定仕様

### 9.1 基本方針
- LLMはオープンソースモデルをAPIサーバとして起動する
- バックエンドはそのLLM APIを呼び出して違和感判定を行う
- LLMは違和感判定専用とし、クイズ判定には使わない

### 9.2 LLM入力情報
バックエンドはLLMに最低限以下を渡す。

- 問題の anomalyId
- anomalyAnswerCanonical
- anomalyAnswerAliases
- プレイヤーの anomalyAnswer

### 9.3 LLM出力仕様
LLMには JSON 形式で以下を返させる。

{
  "isMatch": true,
  "confidence": 0.91,
  "reason": "ユーザー回答はタイトルの誤字を指している"
}

必要項目:
- `isMatch`: boolean
- `confidence`: number
- `reason`: string

判定ルール:
- `isMatch = true` なら違和感正解
- `isMatch = false` なら違和感不正解

### 9.4 タイムアウト・失敗時
- LLM応答タイムアウトは 2 秒以内を目標とする
- タイムアウト、接続失敗、JSON不正、空レスポンス時は不正解扱い
- リトライは行わない
- フォールバック判定は行わない

## 10. 非機能要件

### 10.1 性能要件
- 1回の判定は2秒以内を目標とする
- 「x問」「0問」の表示時間は3秒固定とする
- 同時接続は10人程度を想定する
- 大規模スケーリングは不要

### 10.2 対応環境
- Webブラウザで動作すること
- PCブラウザ前提
- スマホ対応は不要

### 10.3 保存要件
- ユーザのログイン情報は保持しない
- スコア、プレイ履歴、回答履歴は保存しない
- 問題マスタデータのみ保存する

### 10.4 可用性要件
- 判定に失敗した場合でもリカバリ処理は行わない
- LLM判定のブレに対する補正は行わない

## 13. 実装ルール

- 問題出題ロジックはフロントから独立して実装すること
- LLM呼び出し部分は別サービス層として切り出すこと
- クイズ判定ロジックと違和感判定ロジックを分離すること
- 問題追加機能は本体ゲームと疎結合にすること
- 問題データの更新だけで問題追加可能にすること
- UIはシンプルでよく、演出は最小限とする

## 14. 受け入れ条件

以下を満たしたら MVP 完成とする。

1. ホーム画面からゲームを開始できる
2. ランダムに8問連続で出題できる
3. 1プレイ中に正解済み問題が再出題されない
4. クイズ回答と違和感回答を同時入力・同時送信できる
5. クイズ回答が完全一致で判定される
6. 違和感あり問題で、LLMにより違和感判定が行われる
7. 違和感なし問題で、違和感欄に入力があれば不正解になる
8. 正解時に「x問」を3秒表示して次へ進む
9. 不正解時に「0問」を3秒表示して1問目に戻る
10. 8問正解で「クリア!!」画面を表示できる
11. クリア画面からホームへ戻れる
12. 管理ページから問題画像と問題データを追加できる
13. 問題IDが自動生成される
14. タグが自動付与される
15. プレイ履歴を保存しない
